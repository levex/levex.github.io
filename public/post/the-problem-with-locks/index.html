Thoughts on Locks (Part I)

08/26/2016

In this article we will walk through some common mistakes with regards to locking and why spinlocks and mutexes suck in general. We will discuss some alternatives and make a case for lock-less programming. Let’s dive in, it’s going to be interesting.

Consider the following piece of simple code using Linux-kernel style locks: 

void
some_function()
{
    spin_lock(&some_global_lock);
    // some critical section magic
    spin_unlock(&some_global_lock);
    return;
}

At first the code looks great, some structure is protected using a spin_lock_t. However, there is a big problem. The said structure is locked via a global lock, namely some_global_lock. What if the critical section inside the lock calls code that locks the lock again? You get a dreaded deadlock, oh no!

The programmer comes up with a “brilliant” concept: 

void
other_function()
{
    if(is_spin_locked(&some_global_lock))
        spin_lock(&some_global_lock);
    // another critical section
    spin_unlock(&some_global_lock);
    return;
}
Can you spot the problem? How do you detect whether the lock was set when you entered the function in order to decide whether to unlock the spinlock or leave it alone? So, naturally the next thing that comes into mind is to increment some internal counter everytime spin_lock is called by the same thread, and decrement it each time spin_unlock is called. So, we arrive at this code:

void
other_function()
{
    spin_lock(&some_global_lock);
    // another critical section
    spin_unlock(&some_global_lock);
    return;
}



void
some_function()
{
    spin_lock(&some_global_lock);
    // some critical magic
    other_function();
    spin_unlock(&some_global_lock);
    return;
}
Awesome, one may think! We can call both other_function and some_function without having to worry about locks! That’s epic! No. These are called recursive locks. Please avoid them. They are wrong. Just no.

So, let’s discuss why the fact that using recursive locks is so bad. First and foremost, the most crucial in mutual exclusion is the ability to run most of the code in a parallel setting. Recursive locks do not encourage that. In fact, they encourage you to be lazy and just lock everything you need. That sort of encouragement essentially destroys what you wanted in the first place, that is your code won’t be parallel anymore. Instead, a well-behaved programmer would do the following:

void
other_function_unsafe()
{
    // critical section
}



void
other_function()
{
    spin_lock(&some_global_lock);
    other_function_unsafe();
    spin_unlock(&some_global_lock);
}



void
some_function()
{
    spin_lock(&some_global_lock);
    // some critical magic
    other_function_unsafe();
    spin_unlock(&some_global_lock);
}
… which is a much nicer solution. We have the critical section thoughtfully separated, and if we happen to need the same code while under the same lock, we can reuse the code worry-free. This way of working helps you enforce one of the most critical rules of threaded-programming: Keep critical sections as short as possible.

Our solution is still far from perfect. What if Programmer B came along and added a function call into other_function_unsafe? This shouldn’t be a problem per-se, but what if that function goes ridicolous and calls a dozen others? That would again violate the rule we proposed earlier, not to mention what if one of the functions that are called begins to sleep? This is a deadlock waiting to strike. Even if we are lucky to avoid the deadlock, we still have to account for starvation of other threads waiting on this lock.

Obviously, the use of a global lock is a terrible practice as well, because probably you started using threads as a form to implement to concurrency, but using a single global lock does much damage to it, if not completely preventing it. In the next part, we will cover fine-grained locking and all of its benefits.
